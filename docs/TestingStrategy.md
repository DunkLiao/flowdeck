# 測試策略與無障礙效能規範 (Testing Strategy)

本專案採用手動功能驗證為主，搭配整合的自動化邊緣案例測試頁。

## 1. 驗證策略 (Verification Strategy)

### 1.0 本地雙擊啟動驗證
正式使用入口是直接開啟專案根目錄的 `index.html`，不得要求使用者先啟動 Python、本機 HTTP server 或外部網路。

- **測試點 1**：關閉網路後，在檔案總管雙擊 `index.html`。
- **測試點 2**：確認網址列為 `file://`，且首頁、側邊欄、按鈕與 SVG 圖示正常顯示。
- **測試點 3**：新增看板、欄位與卡片後重新整理，確認資料仍存在。
- **測試點 4**：檢查 DevTools Network，不應出現 Google Fonts、Lucide CDN 或其他外部請求。
- **預期結果**：Flowdeck 可完整使用，不需要 Python、Node.js、npm、localhost 或網路。

### 1.1 資料庫 CRUD 單元驗證
在專案根目錄下建立 `test/index.html` 測試檔案，引入 `src/core/db/KanbanDB.js`，實作一鍵測試：
- **測試點 1**：清空並初始化資料庫。
- **測試點 2**：新增一個看板，寫入兩個欄位，分別寫入五張卡片。
- **測試點 3**：刪除一個欄位，檢查底下卡片是否級聯刪除 (Cascading Delete)。
- **測試點 4**：匯出 JSON，確認 Schema 結構與 `exportedAt` 欄位存在。

### 1.2 手動整合測試 (Manual Integration Test Cases)
- **卡片拖曳排序**：
  1. 拖動卡片 A 到卡片 B 之前，放開後重新整理網頁，順序應維持不變。
  2. 拖動卡片 A 到另一個欄位 C，放開後重新整理網頁，卡片應在新欄位 C 內。
- **篩選器連動**：
  1. 篩選優先權為「高」，看板內只剩高優先權卡片。
  2. 此時在搜尋框輸入無關關鍵字，所有卡片應隱藏，顯示「無符合條件之卡片」的空白狀態（Empty State）。

## 2. 邊緣與異常案例 (Edge & Failure Cases)

### 2.1 匯入損毀的 JSON
- **測試方式**：故意匯入缺少 `boards` 鍵值，或是 UUID 重複的 JSON 檔案。
- **預期結果**：系統拋出錯誤，Toast 顯示「匯入失敗：檔案格式不符合看板備份 Schema」，且資料庫不寫入，保持原樣。

### 2.2 長時間閒置與大量資料效能
- **測試方式**：建立 50 個欄位，每欄建立 100 張卡片（共 5000 張卡片）。
- **預期結果**：DOM 渲染時應使用 `requestAnimationFrame` 分批進行或懶渲染（Lazy Render），拖曳卡片時畫面不得產生明顯卡頓。

### 2.3 還原初始設定
- **測試方式**：
  1. 建立第二個看板，並新增卡片、標籤與待辦項目。
  2. 點擊「還原初始設定」後先按取消，確認所有資料仍存在。
  3. 再次點擊「還原初始設定」並確認。
  4. 等待頁面重新載入後檢查側邊欄與看板內容。
- **預期結果**：確認還原後只剩一個「我的工作看板」，欄位為「待處理」、「進行中」、「已完成」，卡片、標籤與待辦項目皆清空；重新整理頁面後仍維持初始狀態。

### 2.4 深淺色切換
- **測試方式**：
  1. 開啟首頁，確認預設為深色模式，側邊欄按鈕顯示「淺色模式」。
  2. 點擊「淺色模式」，確認整體背景、玻璃面板、卡片與輸入框切換為淺色外觀，按鈕顯示「深色模式」。
  3. 重新整理頁面。
  4. 再次點擊「深色模式」切回深色外觀。
- **預期結果**：主題切換不影響看板資料、搜尋篩選、匯入匯出或還原初始設定；重新整理後會維持最後選擇的主題。

### 2.5 RWD 與手機觸控操作
- **測試方式**：
  1. 以桌面寬度開啟 `index.html`，確認側邊欄固定顯示、頂部工具列維持單列、欄位與卡片拖曳維持原本行為。
  2. 將 viewport 調整為 900px 以下，確認左上角顯示看板選單按鈕，側邊欄預設隱藏，點擊後以抽屜方式滑入。
  3. 點擊側邊欄遮罩、按下 `Esc`，以及切換任一看板後，確認抽屜側欄會關閉。
  4. 將 viewport 調整為 640px 以下，確認搜尋框滿版、篩選按鈕不溢出、欄位可水平滑動，單一欄位寬度不超出螢幕。
  5. 在手機或 DevTools 觸控模式下長按卡片，拖曳到同欄不同位置與另一欄後放開，重新整理確認卡片順序與欄位保存。
  6. 在手機或 DevTools 觸控模式下長按欄位標題，拖曳欄位重新排序後放開，重新整理確認欄位順序保存。
  7. 開啟卡片詳細視窗，確認 Modal 在手機寬度下為單欄可滾動，標題、描述、欄位、優先權、到期日、標籤、待辦與刪除按鈕皆可操作。
- **預期結果**：桌面版不退化；手機版不出現主要 UI 水平溢出；抽屜側欄、搜尋篩選、Modal、卡片拖曳與欄位拖曳均可使用且重新整理後保留資料。

## 3. 無障礙 (Accessibility) 與效能規範

### 3.1 無障礙設計 (a11y Checklist)
- **鍵盤友善 (Keyboard Navigation)**：
  - 卡片編輯 Modal 開啟時，焦點應自動鎖定至 Modal 內 (Focus Trap)。
  - 按下 `Esc` 鍵應隨時能關閉當前開啟的彈出視窗 (Modal)。
- **螢幕閱讀器與語義化**：
  - 拖曳元件加上 `aria-grabbed="false"` 狀態屬性，拖曳時切換為 `true`。
  - 所有圖標按鈕（如刪除、新增）必須設有 `aria-label`（如 `aria-label="刪除看板"`）。

### 3.2 效能優化 (Performance Plan)
- **防抖搜尋 (Debounced Search)**：
  - 搜尋欄輸入監聽器加上 150ms 的防抖 (Debounce) 處理，避免每輸入一個字母就觸發整頁 DOM 的重排與匹配。
- **事件委派**：
  - 看板區與側邊欄僅各綁定一個 `click` 事件監聽器，避免大量卡片建立時產生數百個事件監聽器。
